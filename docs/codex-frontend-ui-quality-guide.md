# Codex 프론트엔드 UI/UX 품질 가이드

## 개요

이 문서는 `front-vibe` 같은 내부형 업무관리툴을 Codex로 구현할 때, 기능 구현만 끝내지 않고 실제 제품 수준의 UI 품질까지 끌어올리기 위한 작업 기준이다.

이 프로젝트에서 반복적으로 문제 되는 항목은 아래와 같다.

- 기능은 되지만 디자인이 레거시 관리자 템플릿처럼 보임
- 사이드바, 모달, 테이블, 칸반 같은 핵심 UI가 구현 위주로 끝남
- 줄 깨짐, overflow, spacing 불일치, 지나치게 큰 컴포넌트 밀도 문제가 남음
- hover, focus, disabled, loading, empty, error 상태가 빠짐
- 모바일이나 축소 상태에서 사이드바, 툴바, 상세 패널이 자연스럽게 접히지 않음
- 클릭과 드래그, 상세 진입과 액션 버튼의 상호작용 경계가 모호함

즉, 단순히 “컴포넌트를 만들었다”와 “운영 가능한 제품 UI가 됐다”는 전혀 다르다.

---

## 왜 품질이 쉽게 무너지는가

예를 들어 아래처럼 요청하면:

> 사이드바 만들고, 모달 만들고, 반응형 처리해줘.

Codex는 보통 다음 기준으로 최적화된다.

- 사이드바가 열리고 닫히면 됨
- 모달이 뜨고 닫히면 됨
- 반응형은 화면이 줄어들 때 대충 버티면 됨
- UX 품질 기준이 명시되지 않았으니 최소 구현으로 마무리

이 경우 결과는 거의 항상 `기능 충족형 UI`가 된다.

따라서 요청은 “무엇을 만들지”만 적으면 안 되고, **무엇을 실패로 볼지**까지 같이 적어야 한다.

---

## 이 프로젝트의 UI 방향

현재 `front-vibe`는 다음 조건을 가진다.

- 공개 서비스보다 로그인 후 쓰는 내부 협업툴에 가깝다.
- 핵심 도메인은 `업무`, `검토`, `멤버`, `역할/권한`, `마일스톤`이다.
- `테이블`, `칸반`, `캘린더`, `차트`, `간트`는 같은 데이터를 다른 뷰로 표현하는 구조다.
- `업무 클릭 -> 상세 확인`, `검토 진입`, `드래그앤드롭`, `필터`, `상태 변경`, `모달 액션`이 핵심 상호작용이다.
- 디자인 목표는 마케팅 사이트가 아니라 `잔디`, `두레이`, 현대적인 B2B SaaS 운영툴에 가까운 밀도와 위계다.

### 백엔드/도메인 기준 원칙

- 프론트 수정은 항상 **현재 보이는 화면만** 기준으로 하면 안 된다.
- 앞으로 추가될 `업무` 도메인 기능과 `검토`, `멤버`, `역할/권한`, `마일스톤`, `감사 로그` 같은 타 도메인 로직까지 함께 고려해야 한다.
- 즉, 화면 하나를 고칠 때도 이후 도메인 확장 시 같은 레이아웃/모달/필터/탭 구조를 재사용할 수 있는지를 먼저 본다.
- API와 필드 구조의 기준은 백엔드 저장소 `be-api-server`의 `dev` 브랜치다.
- 프론트에서 새로운 목록, 상세, 모달, 필터를 추가할 때는 가능하면 먼저 `dev` 브랜치 기준 응답 필드와 도메인 모델을 확인하고 맞춘다.
- mock 데이터는 임시 표현일 뿐이고, 최종 구조는 항상 백엔드 `dev`를 기준으로 수렴해야 한다.

권장 기준:

- 백엔드 저장소: [be-api-server](https://github.com/Final-final-team/be-api-server)
- 기준 브랜치: `dev`
- 프론트 설계 원칙:
  - 새 UI는 미래 업무 도메인 기능 확장을 막지 않아야 함
  - 타 도메인과 공통으로 쓸 모달, 표, 상세, 필터 구조를 우선 설계
  - mock 표현보다 실제 API 매핑 가능성을 우선 고려

따라서 이 프로젝트에서는 아래와 같은 UI가 실패로 간주된다.

- 둥근 박스가 화면 대부분을 차지하는 카드형 레이아웃
- 실제 도메인 데이터보다 헤더, 장식, 설명 블록이 더 많은 화면
- 클릭, 드래그, 링크 액션의 경계가 불분명한 행 구조
- 사이드바를 줄이면 레이아웃이 찌그러지거나 텍스트가 세로로 깨지는 구조
- 상세 패널이 메인 테이블을 과하게 침범해서 한눈에 보기 어려운 구조
- 영문 UI가 섞여 사용자 흐름을 방해하는 화면

## 핵심 해결 전략

### 1. 컴포넌트 요청이 아니라 UI 품질 계약서를 준다

Codex에게는 “사이드바를 만들어라”가 아니라 “어떤 UI 품질이 나와야 하고 무엇이 실패인지”를 함께 전달해야 한다.

이 프로젝트에 맞는 기준은 아래와 같다.

```text
당신은 단순 구현자가 아니라 시니어 프론트엔드 엔지니어 + 프로덕트 디자이너 역할이다.

이번 작업 목표:
- 기능 구현이 아니라 운영 가능한 UI 품질까지 포함해 완료
- 내부형 협업툴/업무관리툴 수준의 밀도와 위계를 유지
- 테이블, 칸반, 캘린더, 차트, 간트가 같은 데이터 모델을 공유해야 함
- 데스크톱/태블릿/모바일에서 구조가 자연스럽게 접히고 깨지지 않아야 함
- 레거시 관리자 템플릿처럼 보이면 실패

반드시 지킬 UX 원칙:
- 실제 도메인 데이터가 화면의 주인공이어야 함
- 상단 헤더와 컨텍스트 영역이 실제 표/보드보다 더 큰 비중을 차지하면 실패
- 사이드바는 펼침/접힘 모두 usable해야 하고, 접을 때 레이아웃이 찌그러지면 실패
- 드래그앤드롭은 클릭 액션과 충돌하지 않아야 함
- 일반 클릭은 상세 진입, 길게 누르기나 핸들 잡기는 드래그로 분리
- 모달은 ESC 닫기, 바깥 클릭 닫기, 포커스 트랩, 스크롤 잠금 지원
- 긴 텍스트/버튼 라벨/테이블 셀은 적절한 truncate 또는 wrap 전략 사용
- 모바일에서 horizontal scroll이 기본값처럼 생기면 실패
- hover/active/disabled/loading/empty/error state를 모두 구현
- spacing은 4px 기반 scale로 통일
- heading/body/caption의 타이포 계층을 명확히 구분
- 한 화면에서 primary action은 1개만 가장 강하게 강조
- 화면상 최종 사용자용 텍스트는 기본적으로 한글 사용
- 접근성: aria, keyboard navigation, contrast 고려

시각적 원칙:
- 과한 박스 중첩, 둥근 카드 남발 금지
- 플랫한 리스트/라인/분할 패널 중심 레이아웃 우선
- 보더, 배경, 그림자는 최소한으로 사용
- 모달은 좁고 답답한 세로 패널보다 넓고 읽기 쉬운 가로형 우선

완료 조건:
- 구현 후 스스로 UI 리뷰 수행
- 레이아웃 깨짐, spacing 불일치, 시각적 계층 문제, UX 누락을 찾아 수정
- 변경 이유를 UX 관점으로 요약
```

### 2. 기능 구현 직후 UX/시각 검증을 강제한다

구현 직후 아래 항목을 강제로 점검해야 한다.

- 사이드바를 줄였을 때 아이콘, 텍스트, 검색, 프로젝트 선택이 자연스럽게 접히는가
- 상단 헤더가 실제 도메인 내용보다 더 많은 높이를 먹지 않는가
- 테이블/칸반/캘린더/차트/간트가 모두 같은 시각 톤과 밀도를 유지하는가
- 업무 행 클릭과 드래그앤드롭이 충돌하지 않는가
- 모달이 너무 좁거나 세로로 길게 늘어지지 않는가
- 우측 패널이나 상세 영역이 메인 리스트를 지나치게 밀어내지 않는가
- 모든 visible UI 텍스트가 한글로 정리되어 있는가

### 3. 역할을 분리한 리뷰 루프를 추가한다

가능하면 한 번에 끝내지 말고 아래 루프를 강제한다.

1. `구현`
2. `UI/UX 리뷰`
3. `깨짐/불일치 수정`
4. `스크린샷 또는 실제 화면 재검토`

즉, Codex를 “한 번에 다 맞추는 생성기”처럼 쓰지 말고, **구현자와 리뷰어를 한 번씩 번갈아 강제**해야 한다.

### 4. 가능하면 스크린샷, 디자인 파일, 실제 제품 레퍼런스를 함께 준다

다음 자료가 있으면 품질이 확실히 올라간다.

- 현재 화면 캡처
- 싫은 예시 캡처
- 원하는 제품 레퍼런스
- Figma / FigJam 흐름도

특히 이 프로젝트는 “설명보다 실제 화면” 기준이 잘 먹힌다.  
따라서 아래 같은 피드백은 매우 유효하다.

- “이 박스 너무 크다”
- “도메인 내용이 너무 밑에 있다”
- “사이드바 축소 시 찌그러진다”
- “우측 패널 대신 모달이 더 낫다”
- “색이 아니라 컴포넌트 구조가 문제다”

## 이 프로젝트에서 우선 적용할 실무 원칙

### 1. 화면 위계

- 첫 화면에서 실제 도메인 데이터가 바로 보여야 한다.
- 헤더, 설명, 컨텍스트 바는 최소 높이로 유지한다.
- `업무`, `검토`는 리스트/보드 자체가 첫 번째 시선에 들어와야 한다.

### 2. 레이아웃

- 박스 안에 또 박스를 중첩하는 카드형 레이아웃을 줄인다.
- 플랫한 리스트, 얇은 구분선, 분할 패널, 인라인 요약을 우선 사용한다.
- 우측 고정 패널은 메인 폭을 너무 먹으면 모달이나 오버레이로 바꾼다.

### 3. 상호작용

- 클릭과 드래그는 반드시 분리한다.
- 테이블 행 클릭은 상세 모달, 길게 누르기나 드래그 핸들은 정렬로 연결한다.
- 링크, 버튼, 행 클릭이 서로 이벤트를 먹지 않게 명확히 분리한다.

### 4. 사이드바

- 축소 시 텍스트가 세로로 깨지거나 억지로 압축되면 실패다.
- 접힌 상태와 펼친 상태를 아예 다른 표현으로 설계한다.
- 토글 버튼은 “밖에 떠 있는 장치”보다 문맥 안에 붙어 있는 것이 자연스럽다.

### 5. 모달

- 세로로 길고 답답한 상세 패널보다, 넓고 둥근 가로형 모달이 우선이다.
- 메타 정보, 본문, 액션 영역은 읽는 흐름에 맞게 좌우 분할한다.
- 닫기, ESC, 외부 클릭, 포커스 트랩이 기본이다.

### 6. 뷰 일관성

- `테이블`, `칸반`, `캘린더`, `차트`, `간트`는 전부 같은 데이터 모델과 시각 언어를 공유해야 한다.
- 어떤 뷰든 크기, 타이포, 간격, 상태 라벨이 지나치게 달라지면 실패다.

## 구현 후 필수 체크리스트

- [ ] 실제 도메인 내용이 첫 화면에서 충분히 위쪽에 보인다.
- [ ] 헤더가 콘텐츠보다 더 큰 비중을 차지하지 않는다.
- [ ] 사이드바 접힘/펼침이 둘 다 자연스럽다.
- [ ] 클릭과 드래그가 충돌하지 않는다.
- [ ] 모달이 지나치게 좁거나 세로로 길지 않다.
- [ ] 영어 UI가 남아 있지 않다.
- [ ] hover, active, disabled, loading, empty, error 상태가 있다.
- [ ] 테이블/칸반/캘린더/차트/간트가 같은 제품의 화면처럼 보인다.
- [ ] 모바일/축소 상태에서 layout break가 없다.

## 권장 작업 순서

1. 디자인 토큰과 공통 컴포넌트부터 정리
2. 헤더, 사이드바, 툴바, 모달 같은 공통 크롬 정리
3. `업무`, `검토`처럼 핵심 도메인 화면 먼저 개선
4. 그 다음 `멤버`, `역할/권한`, `마일스톤`으로 톤 확장
5. 마지막에 전체 화면을 다시 돌면서 spacing, line-height, overflow, 언어, 상태를 정리
