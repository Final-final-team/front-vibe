# ADR-0013: 업무 스켈레톤 필드 기준으로 마일스톤 정보 구조 재정렬

- 상태: 채택
- 날짜: 2026-03-17

## 배경

- 2026-03-17 기준 백엔드 `be-api-server`의 task 스켈레톤 브랜치 `origin/feat/24` 에서 `Task` 엔티티는 `authorId`, `title`, `description`, `startDate`, `dueDate`, `status`, `priority`, `createdAt`, `updatedAt` 필드를 가진다.
- 기존 프론트 마일스톤 화면은 진행률과 요약 문구 중심이라, 실제 task 도메인 연동이 시작되면 보여줄 정보 우선순위가 불명확했다.
- 특히 마일스톤 화면은 `연결 업무` 섹션이 중복되고, `최근 변경`도 실제 task 갱신 시각이 아니라 기한을 기준으로 계산하고 있었다.

## 검토한 선택지

### 1. 기존 요약형 마일스톤 화면 유지

장점:
- 수정 범위가 작다.

단점:
- task 도메인 실연동 시 필드 수용력이 약하다.
- 일정/우선순위/최근 갱신 같은 운영 정보가 뒤로 밀린다.
- 사용자가 "지금 무엇이 위험한지" 바로 읽기 어렵다.

### 2. 마일스톤을 운영 요약 + 연결 업무 메타 구조로 재정렬

장점:
- backend task 필드를 자연스럽게 수용할 수 있다.
- `현재 단계 / 위험 / 다음 액션`을 먼저 보여줄 수 있다.
- 연결 업무 목록에서 시작일, 마감일, 최근 갱신, 우선순위를 함께 보여줄 수 있다.

단점:
- mock 데이터와 view-model도 함께 손봐야 한다.

## 결정

마일스톤 화면은 `운영 요약 + 연결 업무 메타` 구조로 재정렬한다.

- 상단: 마일스톤/업무/완료/위험 통계
- 본문 좌측: 마일스톤 목록
- 본문 우측 상단: `현재 단계 / 위험 메모 / 다음 액션`
- 본문 우측 보조영역: 진행률, 범위 요약, 최근 변경
- 하단: 연결 업무별 `상태 / 우선순위 / 담당자 / 시작 / 마감 / 최근 갱신`

또한 mock task 타입에도 backend skeleton 기준 메타(`authorId`, `priority`, `startDate`, `dueDate`, `createdAt`, `updatedAt`)를 반영한다.

## 결과

- 마일스톤 화면이 실제 task 도메인 연동 전에도 어떤 필드를 운영자가 읽게 될지 더 명확해진다.
- task 상세와 milestone 연결 리스트가 같은 필드 체계를 공유할 수 있다.
- 이후 API 연동 시 mock 전용 보정 로직을 줄이고 backend DTO에 더 가깝게 맞출 수 있다.
