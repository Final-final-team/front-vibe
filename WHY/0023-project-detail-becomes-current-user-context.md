# 현재 사용자 문맥은 별도 me API 대신 프로젝트 상세로 푼다

## 질문 배경

- 프론트에서 `내 업무만`, review 액션 가드, 프로젝트 상태 표시를 정확하게 하려면 현재 로그인 사용자의 식별값이 필요했다.
- 하지만 현재 백엔드에는 프론트가 바로 쓰는 전용 `/me` API가 보이지 않는다.
- 반면 프로젝트 상세는 이미 `myUserId`, `membershipStatus`, `status`, `imageUrl`를 제공한다.

## 선택지

### 1. 프론트에서 `/me`가 있다고 가정하고 새 API를 요구한다

- 장점
  - 개념상 단순하다.
  - 전역 사용자 문맥을 프로젝트와 분리할 수 있다.
- 단점
  - 지금 백엔드 계약에 없다.
  - 프론트가 또 임시 가정을 하나 더 만들게 된다.

### 2. 이미 존재하는 프로젝트 상세를 현재 사용자 문맥의 기준으로 사용한다

- 장점
  - 현재 백엔드 API만으로 바로 구현 가능하다.
  - 프로젝트별 멤버십 상태와 사용자 식별을 동시에 얻는다.
  - `내 업무`, task/review 권한 UX, 프로젝트 허브 배지에 바로 활용할 수 있다.
- 단점
  - 프로젝트를 선택한 이후의 문맥이라는 제약이 있다.
  - 전역 사용자 프로필이 필요해지면 나중에 별도 API가 추가될 수 있다.

## 이 프로젝트 기준 추천

- 추천안은 `2번`이다.
- 이 제품은 지금도 프로젝트 허브와 프로젝트 단위 라우트 중심으로 동작하고 있으므로, 프로젝트 상세를 문맥 기준으로 삼는 것이 현재 구조와 가장 잘 맞는다.

## 최종 선택

- `currentUserId`, `membershipStatus`, `project status`는 프로젝트 상세 query를 workspace provider에 편입해서 공급한다.
- 이후 `내 업무만`, review 권한 가드, 프로젝트 상태 표시가 이 값을 우선 사용한다.

## 선택 근거

- 백엔드 `ProjectController`는 프로젝트 상세를 이미 제공한다.
- `ProjectDetailResult`에는 `myUserId`, `membershipStatus`, `status`가 포함되어 있다.
- 프론트가 별도 사용자 API를 임의로 가정하지 않아도, 현재 필요한 UX 대부분을 이 값으로 해결할 수 있다.
