# ADR-0009: 업무 테이블 우선 DnD와 모달 준비형 상호작용

- 상태: 채택
- 날짜: 2026-03-09

## 배경
- 사용자 요청은 현재 범위를 `멤버`, `역할/권한`, `마일스톤`, `업무`, `검토`에만 집중시키는 것이었다.
- 상단의 `사이드킥`, `자동화`, `연동`, 기타 placeholder 액션은 현재 구현 범위를 흐린다.
- 추후 `칸반`, `캘린더`, `차트`, `간트`가 들어올 예정이며, 업무 화면에는 monday.com과 유사한 밀도의 DnD 상호작용이 필요하다.
- `멤버 초대`, `업무 상신`, `검토 처리`는 향후 팝업 형태가 적합할 수 있다는 요구가 있었다.

## 검토한 선택지
### 선택지 A. 업무 테이블에 먼저 DnD를 적용하고 미래 뷰는 비활성 탭으로 남긴다
- 장점:
  - 현재 업무 중심 사용 흐름에 바로 가치를 준다.
  - 이후 칸반/간트 확장으로 이어지기 쉽다.
  - 사용자가 제품 방향을 화면에서 바로 이해할 수 있다.
- 단점:
  - 서버 저장 방식은 뒤늦게 별도 설계가 필요하다.

### 선택지 B. 미래 뷰 구현 전까지 DnD와 팝업 전제 모두 미룬다
- 장점:
  - 초기 구현 범위가 줄어든다.
- 단점:
  - 업무 보드의 핵심 상호작용을 늦추게 된다.
  - 이후 구조 변경 비용이 커질 수 있다.

## 결정
- 상단 셸은 도메인 집중형으로 단순화한다.
- 프로젝트 선택과 현재 도메인 맥락만 유지하고, 범위 밖 액션은 제거한다.
- 미래 뷰는 숨기지 않고 비활성 탭으로 남긴다.
  - `칸반`, `캘린더`, `차트`, `간트`
- 첫 번째 DnD 적용 지점은 업무 테이블이다.
- 같은 마일스톤 그룹 내부에서 row 순서 재정렬만 허용한다.
- DnD 라이브러리는 `dnd-kit`를 사용한다.
- 상신/처리/초대 플로우는 전부 즉시 모달화하지 않지만, 공통 `Dialog`를 활용할 수 있는 구조를 유지한다.

## 결과
- 업무 테이블은 이후 칸반/캘린더/간트로 확장되는 상호작용 기준점이 된다.
- DnD 결과를 서버에 어떻게 저장할지에 대한 계약은 별도 ADR 또는 API 설계가 필요하다.
- 기능별로 모달과 상세 페이지의 책임 분리는 후속 합의가 필요하다.
- 이후 실제 통신에서는 다음 항목을 반드시 정해야 한다:
  - 순서 값 저장 방식
  - 낙관적 업데이트 여부
  - 동시 수정 충돌 처리
  - 실패 시 롤백 UX
  - 실시간 동기화 필요 여부
