# 0019. 프로젝트 허브를 메인 페이지로 사용

## 질문 배경
- 사이드바의 집 아이콘을 눌렀을 때 여러 프로젝트 카드가 보이는 화면이 사실상 메인 페이지여야 한다.
- 사용자는 먼저 프로젝트를 고른 뒤 업무/검토 보드로 들어가는 흐름을 기대한다.

## 선택지
### 1. 메인을 바로 특정 프로젝트 업무 보드로 보낸다
- 장점
  - 구현이 단순하다.
  - 프로젝트가 하나뿐일 때는 빠르다.
- 단점
  - 사용자가 현재 어떤 프로젝트 문맥에 들어왔는지 이해하기 어렵다.
  - 여러 프로젝트를 다루는 확장 방향과 맞지 않는다.

### 2. 메인을 프로젝트 카드 허브로 둔다
- 장점
  - 집 아이콘의 의미가 분명해진다.
  - 프로젝트 선택 후 업무/검토로 들어가는 정보 구조가 자연스럽다.
  - 이후 실제 프로젝트 목록 API가 생겨도 그대로 확장하기 쉽다.
- 단점
  - 허브 페이지와 카드 데이터가 추가로 필요하다.

## 이 프로젝트 기준 권고
- 현재는 `2. 프로젝트 카드 허브를 메인 페이지로 두는 방식`이 맞다.
- 이유는 사이드바 홈의 의미가 분명해지고, 온보딩 이후 진입 동선도 더 자연스러워지기 때문이다.

## 최종 선택
- 루트 진입과 OAuth 콜백 완료 후 기본 진입 화면을 `/projects`로 둔다.
- 집 아이콘과 홈 메뉴는 모두 `/projects`로 연결한다.
- 프로젝트 카드를 눌렀을 때 각 프로젝트의 업무/검토 화면으로 들어간다.

## 선택 근거
- 현재 프로젝트 API가 완전하지 않아도 프론트 정보 구조는 미리 안정화할 수 있다.
- 허브 화면은 이후 실제 프로젝트 목록 API로 교체하더라도 의미가 유지된다.
