# ADR-0007: 로컬 연동용 WSL 런타임과 Vite 프록시

- 상태: 채택
- 날짜: 2026-03-09

## 배경

- 실제 배포 환경이 Linux 계열이므로, 로컬도 가능한 한 같은 조건으로 맞춰 백엔드와 프론트를 실연동 테스트하고 싶었다.
- 현재 백엔드는 기본 설정상 `localhost:5432` PostgreSQL과 `8080` HTTP 서버를 전제하고 있고, 프론트는 `Vite` 개발 서버에서 동작한다.
- 브라우저에서 프론트와 백엔드를 직접 다른 origin으로 연결하면 CORS 설정이 추가로 필요하다.

## 검토한 선택지

### 1. Windows에서 백엔드 실행 + 프론트는 기존 방식 유지

장점
- 바로 실행하기 쉽다.
- 초기 세팅 비용이 가장 낮다.

단점
- 실제 배포 환경과 다르다.
- 파일 경로, 바이너리, optional dependency, Node native module 차이로 재현성이 떨어질 수 있다.
- 이미 한 번 Windows/WSL 경계에서 Java와 DB 상태가 갈려 진단 비용이 커졌다.

### 2. WSL에서 백엔드만 실행하고 프론트는 Windows에서 실행

장점
- 백엔드 런타임은 Linux에 가깝게 맞출 수 있다.
- 프론트는 기존 설치를 그대로 쓸 수 있다.

단점
- 프론트 `node_modules`와 백엔드 런타임이 서로 다른 OS에 놓여 추후 native dependency 문제가 다시 생길 수 있다.
- “어느 쪽 환경 문제인지”가 다시 섞일 수 있다.

### 3. WSL에서 백엔드와 프론트를 모두 실행

장점
- Linux 기준으로 가장 일관된 개발 환경을 만든다.
- 배포 환경과 가장 비슷한 조건에서 테스트할 수 있다.
- Java, PostgreSQL, Node native dependency를 한 OS 기준으로 맞출 수 있다.

단점
- 초기 세팅 비용이 가장 크다.
- WSL에 Java, PostgreSQL, Node 런타임과 프론트 의존성을 다시 맞춰야 한다.

## 프론트엔드-백엔드 연결 방식 선택지

### A. 프론트에서 `http://127.0.0.1:8080`으로 직접 호출

장점
- 설정이 단순해 보인다.
- 실제 API origin이 명확하다.

단점
- 브라우저 기준 cross-origin 요청이므로 CORS 설정이 필요하다.
- 현재 백엔드에는 별도 CORS 설정이 보이지 않는다.
- 프론트 로컬 개발 중에는 매번 CORS를 함께 관리해야 한다.

### B. Vite dev server에서 `/api`를 백엔드로 프록시

장점
- 브라우저는 같은 origin(`127.0.0.1:5173`)으로 요청하므로 CORS 문제가 없다.
- 프론트 코드 수정이 최소다.
- 개발 중 백엔드 origin을 바꿔도 프론트 코드보다 dev server 설정만 바꾸면 된다.

단점
- 개발 환경 전용 설정이 하나 더 생긴다.
- 운영 배포 경로와는 별개의 프록시 레이어가 있다.

## 권고안

- 런타임은 `WSL에서 백엔드와 프론트를 모두 실행`하는 쪽이 맞다.
- 연결 방식은 `Vite proxy`가 맞다.

## 결정

- `WSL + Java 17 + PostgreSQL + Node` 기준으로 실행한다.
- 프론트는 `127.0.0.1:5173`, 백엔드는 `127.0.0.1:8080`으로 띄운다.
- 프론트에서 `/api` 요청은 `vite.config.ts`의 proxy를 통해 백엔드로 전달한다.
- `VITE_API_BASE_URL`은 로컬 개발 기준 `http://127.0.0.1:5173`를 사용한다.

## 결과

- 사용자가 배포 환경과 맞춘 Linux 기준 개발을 원했다.
- 실제로 WSL에서 Java/PostgreSQL/Node를 모두 맞춘 뒤 조회 API와 쓰기 API가 정상 응답하는 것을 확인했다.
- 백엔드에 CORS 설정이 없으므로, 개발 중 브라우저 연결은 direct cross-origin보다 Vite proxy가 더 안전하고 단순하다.
- 현재 repo가 `Vite` 기반이므로 `Next.js` 전환이나 별도 dev gateway 도입보다 비용이 낮다.
