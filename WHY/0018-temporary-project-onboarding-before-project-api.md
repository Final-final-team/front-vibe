# 0018. 프로젝트 API 전 임시 온보딩

## 질문 배경
- 로그인 이후 바로 들어갈 프로젝트 생성/선택 플로우가 없다.
- 현재 백엔드에는 프로젝트 생성/목록 API가 없고, 사용자가 이미 하나의 프로젝트 멤버라는 가정만 존재한다.

## 선택지
### 1. 프로젝트 API가 생길 때까지 아무 화면도 두지 않는다
- 장점
  - 구현이 가장 단순하다.
- 단점
  - 로그인 직후 사용자가 왜 바로 특정 프로젝트로 들어가는지 이해하기 어렵다.
  - 첫 진입 UX가 비어 보인다.

### 2. 임시 온보딩을 만들고 로컬 저장만 한다
- 장점
  - 첫 진입 흐름을 설명할 수 있다.
  - 실제 생성 API가 없다는 점을 숨기지 않으면서 UX 공백을 줄일 수 있다.
  - 나중에 진짜 프로젝트 생성 API가 생기면 교체가 쉽다.
- 단점
  - 입력값이 서버에 저장되지 않는다.
  - 완전한 프로젝트 생성 경험은 아니다.

## 이 프로젝트 기준 권고
- 현재는 `2. 임시 온보딩을 만들고 로컬 저장만 하는 방식`이 맞다.
- 이유는 백엔드 계약을 거짓으로 흉내 내지 않으면서도, 첫 로그인 후 진입 흐름을 설명할 수 있기 때문이다.

## 최종 선택
- `/onboarding/project` 화면을 추가한다.
- 프로젝트 이름과 설명은 브라우저 로컬 저장소에만 저장한다.
- 온보딩 완료 후에는 `VITE_DEFAULT_PROJECT_ID` 기준 기본 프로젝트 업무 화면으로 이동한다.

## 선택 근거
- 백엔드 task/review API는 프로젝트 멤버십이 이미 있는 상태를 전제로 한다.
- 초대 기능과 프로젝트 생성 API가 없으므로, 온보딩에서 실제 생성 성공을 가장하면 오히려 계약이 어긋난다.
- 로컬 저장 기반 임시 온보딩은 UI 공백을 메우면서도 나중에 실제 API로 자연스럽게 교체할 수 있다.
