# 0017. OAuth 전용 진입과 단일 프로젝트 가정

## 질문 배경
- 백엔드 기준으로 실제 OAuth 로그인만 허용하고 싶다.
- 현재는 초대 기능이 없고, 로그인한 회원이 이미 하나의 프로젝트에 속해 있다고 가정한다.

## 선택지
### 1. 프론트에 로컬 로그인과 OAuth를 같이 둔다
- 장점
  - 로컬 테스트가 빠르다.
  - Google Console 설정이 덜 갖춰져도 개발이 가능하다.
- 단점
  - 실제 운영 로그인 흐름과 UI가 달라진다.
  - 인증 관련 버그가 운영 직전까지 숨어 있을 수 있다.
  - 사용자가 어떤 로그인 방식을 써야 하는지 혼란이 생긴다.

### 2. 프론트를 OAuth 전용으로 잠근다
- 장점
  - 실제 백엔드 운영 계약과 화면 흐름이 동일하다.
  - 로그인 성공 후 쿠키 발급, 콜백, 진입 가드까지 한 번에 검증할 수 있다.
  - 인증 UX가 단순하다.
- 단점
  - Google OAuth 설정이 안 되어 있으면 로컬 개발이 불편하다.
  - 테스트 계정 준비가 필요하다.

## 이 프로젝트 기준 권고
- 현재는 `2. 프론트를 OAuth 전용으로 잠그는 방식`이 맞다.
- 이유는 이 프론트가 백엔드 계약을 그대로 따라야 하고, 사용자가 실제로 마주할 진입 경험도 운영 방식과 같아야 하기 때문이다.

## 단일 프로젝트 가정
- 백엔드 task/review API는 프로젝트 멤버십을 전제로 동작한다.
- 초대 플로우가 아직 없으므로, 로그인 성공 후 사용자가 이미 특정 프로젝트의 ACTIVE 멤버라고 가정한다.
- 프론트는 `VITE_DEFAULT_PROJECT_ID`를 기준으로 로그인 직후 진입 프로젝트를 고정한다.

## 최종 선택
- 로그인 화면은 OAuth 전용으로 유지한다.
- 비로그인 사용자는 무조건 `/login`으로 보낸다.
- 로그인 성공 후에는 기본 프로젝트 업무 화면으로 이동한다.

## 선택 근거
- 백엔드 `SecurityConfig`는 OAuth 로그인과 쿠키 bearer 인증을 기준으로 설계되어 있다.
- 업무 API는 프로젝트 멤버십 검사를 통과해야 접근 가능하다.
- 초대/프로젝트 선택 기능이 아직 없으므로, 단일 프로젝트 진입 가정이 현재 구현 범위에서 가장 일관적이다.
