# 단일 프로젝트 셸 유지 이유

## 질문 배경
- 최신 백엔드는 task/review/auth/consent API는 제공하지만, 프로젝트 목록을 내려주는 REST API는 아직 없다.
- 프론트 셸은 프로젝트 선택기를 전제로 설계되어 있다.

## 선택지
### 1. 프로젝트 선택 UI를 유지하고 mock 프로젝트 목록을 계속 사용한다
- 장점: 기존 셸 변화가 적다.
- 단점: 백엔드 기준이라는 원칙을 깨고, 없는 데이터를 있는 것처럼 보이게 된다.

### 2. 프로젝트 선택 기능을 제거하고 완전 단일 프로젝트 앱으로 단순화한다
- 장점: 백엔드 기준을 지키기 쉽다.
- 단점: 기존 업무관리 셸의 틀을 많이 잃는다.

### 3. 셸은 유지하되 실제 동작은 설정된 단일 프로젝트 하나로 제한한다
- 장점: 기존 UI 틀을 유지하면서도 fake 다중 프로젝트 동작을 줄일 수 있다.
- 단점: 프로젝트 선택기가 사실상 비활성 또는 단일 옵션이 된다.

## 추천
- 현재 단계에서는 3번이 가장 현실적이다.

## 최종 선택
- 실제 데이터는 설정된 단일 프로젝트 id를 기준으로 조회하고, 프로젝트 셸은 단일 항목만 노출한다.

## 판단 근거
- 백엔드에 없는 프로젝트 목록 기능을 프론트가 계속 가짜로 제공하면 이후 전체 라우팅과 권한 흐름이 다시 틀어진다.
- 반대로 셸을 완전히 버리면 사용자가 원한 업무관리 틀을 잃는다.
- 따라서 현재는 단일 프로젝트 기준으로 정렬하고, 추후 프로젝트 API가 생기면 셸 선택기를 다시 확장하는 것이 재작업이 가장 적다.
