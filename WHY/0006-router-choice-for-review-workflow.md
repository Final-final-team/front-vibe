# ADR-0006: 리뷰 워크플로우용 React Router

- 상태: 채택
- 날짜: 2026-03-09

## 배경

리팩터링 과정에서 review 기능을 단일 페이지 모달 구조로 유지할지, 라우터를 도입할지 결정해야 했다.

## 검토한 선택지

### 1. React Router 도입

장점:
- 업무 목록, review 목록, review 상세, 제출/수정 화면을 URL 단위로 분리할 수 있다.
- 브라우저 뒤로가기와 직접 링크 공유가 자연스럽다.
- 내부 업무툴에서도 화면 책임이 명확해진다.

단점:
- 현재 단일 `App.tsx` 데모 구조보다 초기 리팩터링 범위가 커진다.

### 2. 단일 페이지 유지

장점:
- 현재 구조를 덜 바꾼다.
- 작은 데모 수준에서는 빠르게 만들 수 있다.

단점:
- review 도메인처럼 상태와 화면이 많아지면 복잡도가 빠르게 증가한다.
- URL로 상태를 표현하기 어렵다.

## 권고안

`React Router` 도입이 맞다.

## 결정

- 라우팅: `React Router`
- 기본 경로:
  - `/tasks`
  - `/tasks/:taskId/reviews`
  - `/reviews/:reviewId`
  - `/tasks/:taskId/reviews/new`
  - `/reviews/:reviewId/edit`

## 결과

review 기능은 목록, 상세, 편집, 상신이 모두 독립된 화면 책임을 가진다. 이를 단일 페이지 모달로 억지로 감싸기보다 URL 구조로 나누는 편이 구현과 유지보수 모두에서 유리하다.
