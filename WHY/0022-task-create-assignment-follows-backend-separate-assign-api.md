# 업무 생성 시 담당자 지정은 생성 후 할당 API로 분리한다

## 질문 배경

- 프론트의 `새 업무 추가` 모달에는 담당자 개념이 없었지만, 백엔드에는 담당자 할당 기능이 이미 존재했다.
- 다만 백엔드 계약을 확인해 보니 `업무 생성`과 `담당자 할당`은 한 요청이 아니라 별도 API로 나뉘어 있었다.

## 선택지

### 1. 프론트에서 생성 body에 담당자 값을 억지로 넣는다

- 장점
  - 프론트 구현은 단순해 보일 수 있다.
- 단점
  - 백엔드 계약과 어긋난다.
  - 실제 API 요청 형식이 달라져 바로 깨진다.
  - 이후 백엔드와 프론트가 서로 다른 진실을 갖게 된다.

### 2. 생성 성공 후 `assign/me` 또는 `assign`를 연달아 호출한다

- 장점
  - 현재 백엔드와 1대1로 맞는다.
  - 권한 정책도 백엔드가 그대로 최종 판정한다.
  - `권한 없으면 본인만`, `권한 있으면 다른 멤버 지정` UX를 자연스럽게 풀 수 있다.
- 단점
  - 프론트에 한 단계 더 많은 상태 관리가 필요하다.
  - 생성은 성공했지만 할당에서 실패하는 경우를 별도로 처리해야 한다.

## 이 프로젝트 기준 추천

- 추천안은 `2번`이다.
- 이 저장소는 백엔드 구현을 source of truth로 두고 있으므로, 계약을 바꾸는 방식은 선택하면 안 된다.

## 최종 선택

- `새 업무 추가` 모달에는 담당자 선택 UI를 넣되,
- 실제 요청은 `업무 생성` 후 `assign/me` 또는 `assign`를 이어서 호출한다.

## 선택 근거

- 백엔드 `TaskCreateRequest`에는 담당자 필드가 없다.
- 백엔드 `TaskCommandController`는 `POST /tasks/{taskId}/assign` 와 `POST /tasks/{taskId}/assign/me`를 별도로 제공한다.
- 권한도 `TASK_ASSIGN`으로 별도 검증되므로, 프론트는 UI로 보조하되 서버 판정을 최종 진실로 따라야 한다.
