# TanStack Query vs Zustand vs Redux Toolkit

## 질문 배경

사용자가 `Zustand`와 `Redux Toolkit`이 무엇인지, 이 프로젝트에 무엇이 맞는지 질문했다.

## 선택지

### 1. TanStack Query + 로컬 상태

장점:
- 서버 상태, 로딩, 에러, 캐시, 재조회, 무효화를 자연스럽게 다룬다.
- review API처럼 조회/변경이 많은 구조에 적합하다.
- 모달과 폼은 로컬 상태로 분리하면 구조가 단순하다.

단점:
- 전역 UI 상태가 많아지면 별도 스토어가 필요할 수 있다.

### 2. TanStack Query + Zustand

장점:
- 서버 상태는 Query, 전역 UI 상태는 Zustand로 나누기 쉽다.
- Redux보다 가볍고 학습 비용이 낮다.

단점:
- 아직 필요하지 않은 전역 상태까지 미리 만들 가능성이 있다.
- 프로젝트 초기에 구조를 과하게 넓힐 수 있다.

### 3. Redux Toolkit

장점:
- 큰 팀과 복잡한 전역 상태 구조에서 추적성과 규칙성이 좋다.
- 미들웨어와 디버깅 도구가 강하다.

단점:
- 현재 규모와 요구에 비해 무거울 수 있다.
- 보일러플레이트와 설계 비용이 증가한다.

## 이 프로젝트 기준 추천

`TanStack Query + feature 로컬 상태`를 기본값으로 둔다.

## 최종 선택

- 서버 상태: `TanStack Query`
- 폼, 모달, 업로드, 드래프트: 로컬 상태
- `Zustand`와 `Redux Toolkit`은 지금 단계에서 도입하지 않음

## 선택 근거

현재 핵심 복잡도는 "review API 호출과 상태 동기화"이지 "거대한 전역 상태"가 아니다. 따라서 Query가 먼저 해결해야 할 문제에 가장 직접적이고, UI 상태는 로컬에서 관리하는 편이 유지보수가 쉽다.
